# MiniQ-VL 优化指南

本文档给出基于当前实现的优化路线。内容分为“已实现机制”和“建议实验”，避免把规划误写成现有能力。

## 优化原则

每次实验只改变少量变量，并固定：

- 模型初始检查点
- 训练与评估数据切分
- 随机种子
- 生成参数
- 评估提示词
- 指标与人工抽检规则

优先建立小规模、可重复的基线，再扩大数据量与训练规模。

## 已实现机制

### SFT

- LoRA 微调与合并导出
- 视觉和语言模块冻结开关
- DDP、混合精度与梯度累积
- 梯度裁剪
- 梯度检查点
- 定期保存与断点恢复

### DPO

- chosen/rejected 偏好训练
- 仅对助手回答 token 计算序列得分
- 默认使用冻结参考模型
- 可从合并后的 SFT 模型目录初始化

### GRPO

- 每个提示独立生成多条回答
- 组内相对优势标准化
- 裁剪策略目标
- 可选参考模型 KL
- CLIP、LLM Judge、长度及扩展规则奖励
- DDP 样本分片

### 评估

- Base、SFT、DPO 统一比较
- CLIPScore、Self-BLEU、LLM Judge
- 逐样本结果与失败统计

## 建议优化顺序

### 1. 先修正数据质量

训练前抽检图像、提示词和回答是否匹配，并重点检查：

- 图像损坏或无法解码
- 回答为空、截断或模板污染
- 过短、过长和重复描述
- DPO chosen/rejected 差异不明确
- GRPO 数据与奖励目标不一致

数据质量通常比单纯扩大训练步数更重要。

### 2. 建立 SFT 基线

建议先使用小数据集完成端到端验证：

```bash
torchrun --nproc_per_node=1 trainer/train_sft.py \
  --model_path ./model/Qwen3-VL-2B-Instruct \
  --data_path ./dataset/minimind-v_dataset/sft_i2t.parquet \
  --max_samples 1000 \
  --use_lora 1 \
  --epochs 1
```

关注训练损失、生成样例、显存峰值和评估失败率。确认流程稳定后，再调整数据量、LoRA rank、学习率和有效 batch size。

### 3. 优化 DPO 偏好信号

DPO 的关键不是偏好样本数量，而是 chosen 与 rejected 是否形成稳定、可信的质量差异。

建议实验：

| 实验变量 | 建议起点 |
|---|---:|
| `beta` | `0.1` |
| 学习率 | `5e-6` |
| epoch | `1` |
| 参考模型 | 开启 |
| 最大序列长度 | 根据数据长度分布设置 |

若显存不足，可先减小 batch size 与序列长度。关闭参考模型会改变训练目标，不应只作为无差别的显存开关。

### 4. 校准 GRPO 奖励

GRPO 优化前，应先离线检查每个奖励分量的分布：

- 均值、标准差和极端值
- 不同奖励之间的相关性
- 高奖励样本是否真的更好
- Judge 失败时的回退值是否造成偏差

当前默认核心奖励是 CLIP、LLM Judge 和长度奖励。建议先保持扩展奖励权重为 0，确认核心奖励有效后，再逐个启用。

推荐从较小生成组开始：

```text
group_size = 2 或 4
max_new_tokens = 128 至 256
use_ref_model = 0 或在显存充足时开启
```

随后再评估更大生成组是否带来足够收益。

## 显存优化

按优先级尝试：

1. 减小每卡 batch size。
2. 增大梯度累积以维持有效 batch size。
3. 减小最大序列长度与生成长度。
4. 启用梯度检查点。
5. 使用 LoRA 并冻结视觉模块。
6. 降低 GRPO `group_size`。
7. 在明确接受目标变化后，考虑关闭参考模型。

DPO 与启用参考模型的 GRPO 会同时驻留策略模型和参考模型，应提前预留显存。

## 吞吐优化

- 确认训练数据位于高速本地存储。
- 根据 CPU、存储和图像解码速度调整 `num_workers`。
- 在目标硬件上比较 bfloat16 与 float16。
- 控制保存频率，避免高频大检查点阻塞训练。
- GRPO 中优先减少无效 API 请求和过长生成。
- 只有在验证兼容性后再启用 `torch.compile`。

Shell 脚本参数与 Python CLI 默认值并不完全相同，吞吐对比时必须记录最终执行命令。

## 奖励与 API 稳定性

LLM Judge 依赖外部 API。为避免训练奖励被网络状态主导：

- 记录请求失败率、重试次数和延迟。
- 对失败响应设置明确且可追踪的处理方式。
- 小规模缓存或离线预评估 Judge 结果。
- 定期人工检查高分与低分样本。
- 不要只根据单一 Judge 模型决定最终质量。

## 推荐实验矩阵

| 阶段 | 变量 | 对照 |
|---|---|---|
| SFT | LoRA rank、学习率、数据量 | Base 与固定 SFT 基线 |
| DPO | `beta`、偏好数据质量、参考模型 | 同一 SFT 初始化 |
| GRPO | 奖励权重、group size、KL 系数 | 同一 SFT 初始化 |
| 评估 | 单样本质量、多样性、人工抽检 | 固定图像与生成参数 |

每轮实验至少保留：

- 完整启动命令
- Git 提交或代码版本
- 训练日志与关键配置
- 检查点路径
- `eval_all.py` 输出目录
- 人工抽检结论

## 常见问题定位

### 损失正常但生成质量下降

检查标签 mask、提示模板、数据截断和训练数据质量。DPO 还应检查 chosen/rejected 是否颠倒，以及回答区域 mask 是否覆盖正确。

### GRPO 奖励无明显差异

检查同组回答是否真的独立生成、奖励是否大量回退为同一值，以及奖励权重是否被某一分量完全主导。

### 多卡结果异常

检查有效 batch size、数据分片、随机种子、日志聚合和保存进程。先在单卡小数据上验证，再扩展到多卡。

### 评估结果不稳定

固定采样参数与评估集；主要对比可先使用单条生成，另开实验衡量多样性。始终查看逐样本结果与失败统计。

## 当前优先改进项

1. 将 DPO 数据准备脚本中的硬编码参数改为 CLI。
2. 为真实模型加载、单步训练和检查点恢复增加 GPU 冒烟测试。
3. 统一训练入口与通用数据集模块，减少重复实现。
4. 为 GRPO 奖励分量增加结构化监控与缓存。
5. 为预训练阶段补充启动脚本、数据说明和验证流程。
